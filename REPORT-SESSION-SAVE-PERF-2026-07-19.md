## 파동 진행 기록

### 파동 0 — 하니스 (완료)
- packages/session/scripts/click-perf-harness.mts + `perf:clicks` 스크립트.
- 기준선이 진단과 일치: 클릭당 비용·payload가 역사에 비례(6.6×/5.1×).

### 파동 1 — 속효 다이어트 (완료)
- 1a 저널 발산 감시: 기대 '스냅샷' 대신 기대 해시+RNG 위치만 유지 — append당 전체 상태
  문자열화 3→1회, 전체 클론 1회 제거(감시력 등가: 외부 변조 거부 테스트 고정).
- 1b 체크포인트 프리셋 참조화: presetRef {id,version} + 세션 presetSnapshots 사전(변경 시점만 본문).
  저장 history에는 참조된 비현행 프리셋만 동봉. 구형 인라인 체크포인트 하위호환.
- 1c 디스크 undo 5개(RAM 30 유지 — 오너 결정 1).
- 신규 테스트 3(발산 감시·참조 undo 왕복·깊이 분리), 세션 126/126.
- 효과: 1,000이벤트 p95 712→158.6ms(4.5×), payload 30.7→6.1MB. 성장배율은 여전 —
  남은 지배 항(저널 전량 직렬화·integrity 전량 해시)은 파동 2·3 대상.

### 파동 2 — 봉인 에폭 분리 보관 (완료)
- 핫 저장은 `sealedEpochRefs`(offset·sealedIndex·sealHash)만, 본문은 `<세션id>::sealed-epoch:<n>`
  불변 레코드로 봉인 시점 1회 기록. 로드는 `PlaySession.assembleSnapshot`이 본문을 fetch·sealHash
  대조 후 integrity 범위 밖 `sealedEpochBodies`로 동봉 → restore가 참조 검증 후 합체.
- 변조 탐지 3면 테스트: 본문 변조→조립 거부, 참조 변조→integrity 거부, 본문 누락→bodies_missing.
- 구형 인라인 스냅샷 로드→다음 save 승격, 채팅 삭제 GC(레코드 동반 삭제) 테스트 고정. 세션 130/130.
- **봉인 시나리오 실증**: 이벤트 600·에폭 2 회차의 클릭 p95 86.4ms·payload 3.8MB —
  payload가 전체 역사(600)가 아니라 현 에폭(200)에만 비례한다. 내보내기/백업은 여전히 자기완결(인라인).

### 파동 3 — integrity v2 (완료)
- 섹션 해시 합성(meta/messages/engine/memory/journal/history/bindings/extras) + `integrityVersion: 2`.
  저장 시 큰 두 섹션은 롤링 체인 재사용: 저널은 클래스 유지 이벤트 체인+initial 다이제스트+기대 해시
  기반 head(저장마다 전체 상태 재문자열화 제거), 메시지는 append O(1) 연장·그 외 변형 시 무효화.
- 검증은 전체 재계산 대조(로드 1회 O(n)). 커버리지는 v1 초집합(lastLogs 신규 포함).
- **변조 매트릭스**: 8개 섹션 1필드 변조 + 서명 삭제 + v1 다운그레이드 위장 전부 거부(신규 테스트 2).
  구형(버전 없음) 스냅샷은 구 알고리즘으로 계속 검증(하위호환 테스트 유지). 세션 132/132.
- 봉인 시나리오 p95 86.4→79.7ms. 잔여 지배 항은 payload 클론·IDB 구조적 클론 — 파동 4 대상.
- 롱런 카나리아의 에폭 위조 재서명 지점을 v2 인지형으로 갱신(디싱크 0·해시 일치 재확인).

### 파동 4 — 저장 샤딩 + ledger 표시 병합 (완료)
- 메시지(100개)·저널 이벤트(200개) 청크 레코드 분리, 코어는 빈 배열+매니페스트. 완결 청크는
  해시 캐시로 재기록·재해시 생략(불변 검증: updatedAt 동결 테스트). integrity는 '조립된 전체'
  서명이라 청크 변조·누락은 조립 후 검증·명시 오류로 잡힘. 구형 통 레코드 로드→다음 save 승격,
  undo 잘림 고아 청크 정리, 채팅 삭제 GC 확장. 세션 135/135.
- **ledger 표시 병합(오너 결정 3)**: MessageList가 연속 ledger 메시지를 접힌 영수증 묶음으로 렌더
  (저장 병합은 undo 의미론 훼손이라 금지 — 렌더만). e2e 왕복 고정.
- **pending 잠금**: 진행 중 콘솔 클릭이 조용히 버려지던 레이스를 pointer-events 잠금으로 봉인
  (Playwright는 자동 대기, 사용자는 잠금 시각 피드백 — 눌림 피드백의 완성형).
- e2e 보수: IDB를 직접 읽던 management-narration 헬퍼를 샤드 인지형으로, 표시 병합 반영 카운트 2건.

### 파동 5 — 행동 영수증 선저장 + 결과 선표시 (완료)
- 엔진 결과·RNG·저널 위치·상태 해시를 담은 소형 WAL을 먼저 저장하고 결과를 즉시 렌더한다. 큰 본 저장은
  직렬 대기열에서 이어서 처리하며, 다른 UI에서 연속 클릭해도 입력을 버리지 않는다.
- 크래시 복구는 엔진 결과를 정확히 재적용하고, 복구할 수 없는 마지막 LLM 서사만 명시적인 엔진 영수증으로
  대체한다. undo/분기 헤드와 불일치하거나 손상된 WAL은 격리된다.
- `ledgerDeltas`는 앵커별 집계되는 미처리 서사 큐(최대 24그룹·그룹당 표본 3개)로 교체해 silent 플레이 뒤
  AI 서사 모드로 돌아갈 때 수백 건이 한꺼번에 프롬프트로 쏟아지지 않게 했다.

### 파동 6 — 기억 원장 수명 관리 + 체크포인트 구조 공유 (완료)
- 반복 엔진 사실은 원본 이벤트 저널이 영구 보존하므로 기억 검색에는 앵커별 `현재 1개 + 직전 1개`만 둔다.
  약속·관계·비밀·사용자/서사 기억은 이 정리 대상이 아니며 그대로 보존한다.
- RAM undo 체크포인트는 매 클릭마다 기억 전체를 깊은 복사하지 않고 불변 레코드 참조를 공유한다. 원장 변경은
  기존 객체를 고치지 않고 새 객체로 교체하므로 과거 체크포인트의 기억은 변하지 않는다.
- 구형 장기 회차도 복원 시 같은 보관 정책으로 정리된다. 미래 기억 부활 금지, undo/redo, 대안 응답 전환,
  무결성·에폭 왕복 검사를 유지했다.
- **실측:** 1,000이벤트 p95 110.9→2.3ms, payload 4.11MB→97.9KB, 역사 성장배율 7.13→0.86.

## 최종 평가와 잔여

- 엔진 클릭 p95는 기준선 **712.3ms → 2.3ms**, 저장 payload는 **30.7MB → 97.9KB**로 줄었다.
  기억 원장 수술 후 목표(p95 ≤50ms,
  역사 비비례)를 여유 있게 통과했다. 이 수치에서는 별도 기억 샤딩이 복잡도와 복구 위험만 늘리므로 시행하지
  않는다. 다음 잔여는 저장이 아니라 콘솔 원본 이미지 디코드 비용이며 영구 썸네일 단계에서 다룬다.

### 파동 7 — 이미지 계층·대형 목록 가상화 (완료)

- 관리 타일은 192px WebP 영구 캐시, 상세은 필요 해상도, 기지 대표·채팅·확대는 원본을 유지한다.
- 50명을 넘는 목록은 TanStack Virtual로 화면 주변 행만 만든다. 120명 브라우저 시험에서 전체 DOM을 만들지 않으면서 검색 결과와 선택 동작이 유지됐다.
- 에셋 원본·썸네일은 실제 워크로드에서 이미 IndexedDB 지연 캐시로 목표를 만족하므로 별도 OPFS 이미지 팩 복제는 시행하지 않았다.

### 파동 8 — 안전한 다음 뽑기 선준비 (완료)

- `ProjectRuntime.peek`은 상태와 RNG를 복사한 임시 실행만 사용한다. 라이브 상태·주사위·저널·구독 알림을 바꾸지 않으며, 직후 실제 실행 결과와 바이트 동등함을 테스트로 고정했다.
- 현재 상태 해시+RNG 위치를 키로 다음 고용 후보 5명의 썸네일을 유휴 시간에 생성·디코드한다. undo/redo나 다른 행동으로 키가 달라지면 낡은 예측을 쓰지 않는다.
- 브라우저 e2e는 뽑기 클릭 전에 썸네일 저장이 완료되고, 클릭 뒤 5장 모두 로딩 빈칸 없이 표시됨을 검증한다.

### 파동 9 — 연속 행동 WAL 체인과 브라우저 100ms 관문 (완료)

- 각 클릭은 작은 행동 영수증을 먼저 영구 저장한다. 사용자가 연속 클릭하면 영수증을 체인으로 누적하고, 큰 전체 저장은 750ms 유휴 뒤 한 번만 수행한다.
- 종료가 전체 저장보다 빨라도 영수증 3개로 세 행동·세 undo 지점·세 영수증 메시지를 전부 복구하는 테스트를 추가했다. 손상·부모 분기·상태 해시·RNG 불일치는 계속 격리한다.
- 채팅 목록의 턴 숫자 갱신도 1초 유휴 뒤로 보냈다. 게임 결과와 안전 영수증은 먼저 확정되며, 목록 갱신 실패는 세이브 손실로 오인하지 않도록 진단에 따로 남긴다.
- Chromium에서 로컬 엔진 버튼 20회 연속 클릭의 눌림→영수증→결과 페인트 p95 ≤100ms 관문을 3회 반복 통과했다. 1,000사건 하니스는 p95 3.62ms, payload 97.9KB, 성장배율 0.97이었다.

### Worker·저장소 후보 최종 판정

- 세션 저장은 이미 SQLite WASM `OpfsDb`를 전용 Worker에서 사용하고, 실패 시 IndexedDB로 전환한다. 조사안의 “OPFS로 교체”는 현행과 중복이므로 시행하지 않았다.
- 엔진 정본 전체 Worker 이전은 브라우저 100ms 관문과 1,000사건 3.62ms 관문을 모두 통과한 뒤에는 전체 상태 복사·18개 셀렉터 프로토콜·장애 복구 복잡도만 늘릴 가능성이 크다. 합의한 “실측에서 이길 때만 유지” 조건에 따라 시행하지 않았다.
- 설정 화면에는 실제 저장 방식과 4단 복구 안내(이미지 캐시만/에셋 모듈만/사이트 전체/백업 없음)를 표시한다.

## 최종 종합 검증

- **화면 반응:** 모든 엔진 버튼 공통 진입점에서 다음 페인트를 먼저 양보하고, 행동 영수증 저장 직후 엔진
  결과를 렌더한다. 서로 다른 화면의 버튼을 빠르게 이어 눌러도 입력을 버리지 않고 순서대로 처리한다.
- **이미지:** 관리 콘솔은 최대 192px WebP 영구 썸네일을 사용하고 채팅 스프라이트는 원본을 유지한다.
  생성 동시성은 2, 캐시는 2,048장/64MB 상한이며 브라우저 IDB 실저장을 e2e로 확인했다.
- **30일 장기 플레이:** 엔진 사건 608건, 작전 60회 완료, 디싱크 0건, 10·20·30일 저장 왕복 해시 일치,
  같은 시드 재실행 최종 해시 일치. 불변식 위반 0건, 프롬프트 성장비 1.069(허용 1.3 이하).
- **장기 성능:** 1~10일 사건 평균 1.91ms, 11~20일 1.89ms, 21~30일 1.85ms로 누적 악화 없음.
- **전체 검증:** 루트 typecheck/test/build 통과, 브라우저 e2e 56/56 통과. 여기에는 모바일 세로·가로,
  새로고침 저장 복구, 에셋 모듈 재시작 복원, 소녀전선 네이티브 15개 플레이 경로가 포함된다.
