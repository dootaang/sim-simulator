# 플레이 경험 패리티 감사 — 사용자 동작 기준

- 감사일: 2026-07-14
- 비교 기준: 마이그레이션 직전 `03074d8^` ↔ 현행 `895efab`
- 우선 기준: ADR 0001 → ADR 0002 → DESIGN → ROADMAP → BACKLOG
- 판정 기준: 파일이나 함수 이름이 아니라 **사용자가 버튼을 누른 뒤 무엇을 보고, 어디에서 다음 행동을 하는가**

> 이 문서의 1~6절은 WP0 당시의 기준선이다. 이후 WP1~WP6 복구 결과와 확대 감사는 7절부터 기록한다. 과거 판정표를 현재 상태로 오해하지 않는다.

## 1. 결론

엔진 계산과 저장 기반은 복구됐지만, 마이그레이션 전 플레이 경험과 관리 화면의 행동 계약은 아직 부분 복구 상태다.

10개 흐름 중 완전히 동등한 것은 없고, **부분 6개·유실 4개**다. 다만 모두가 마이그레이션 유실인 것은 아니다. 시뮬레이션을 별도 패널로 분리하면서 생긴 이동 문제와 편집기 미리보기의 좁은 배치는 새 Svelte 화면의 설계 부채다.

사용자에게 가장 크게 보이는 단절은 다음 네 묶음이다.

1. 메시지나 현장 행동을 실행해도 채팅이 최신 위치로 이동하지 않고, 시뮬레이션 패널이 열린 채라 응답 대기 상태도 보이지 않는다.
2. 화자와 `outfit` 값은 저장되지만 마지막 이미지 해석 단계에서 버려져, 실제 `silvia_default_0` 같은 파일까지 도달하지 못한다.
3. 고용·시설·의뢰 엔진은 살아 있지만, 옛 셀렉터가 만들던 정원·비용·가능 여부·진짜 성공 확률이 UI에서 사라졌다.
4. 옛 앱은 관리 기능이 채팅 안에 있었지만 현행은 별도 패널이다. 따라서 현장 행동 뒤 채팅 복귀라는 새 연결 계약이 필요한데 구현되지 않았다.

## 3. 흐름별 근거

### 흐름 1 — 사용자가 메시지를 보냄

**옛 동작 (`03074d8^`)**

- 전송 중 입력과 전송 버튼을 잠그고(`app/src/playView.js:475`, `:478`), `생각 중...`을 메시지 목록에 표시했다(`:451` 전후).
- 사용자 입력과 응답으로 다시 그릴 때마다 메시지 목록을 맨 아래로 내렸다(`:308-309`). 따라서 위쪽 기록을 보다가 전송해도 최신 사용자 입력과 응답이 보였다.

**현 동작 (`895efab`)**

- `PlayerPage.svelte:82`가 `busy`와 중단 컨트롤을 관리하므로 전송·대기 자체는 정상이다.
- `PlayerPage.svelte:133`은 데스크톱에서는 `<main>`, 모바일에서는 `.list`를 서로 다른 스크롤 컨테이너로 쓴다(`PlayerPage.svelte:138`). 어느 경로에도 새 메시지 뒤 `scrollTop`, `scrollTo`, `scrollIntoView`가 없다.

**격차와 판정 — 유실.** 최신 입력으로 이동하는 사용자 계약이 사라졌다. WP1에서 “사용자가 전송한 순간”과 “응답이 도착한 순간”을 구분해 하단 이동을 보장해야 한다. 사용자가 일부러 과거 기록을 읽는 중 들어온 비자발적 갱신까지 무조건 끌어내리는 정책은 별도로 피한다.

### 흐름 2 — 시뮬레이션에서 현장 행동을 누름

**옛 동작 (`03074d8^`)**

- 영업·의뢰·하루 마감 등은 채팅 화면 안의 `현장 행동` 접이식 메뉴에서 실행됐다(`app/src/playView.js:908-1037`). 별도 화면이 아니므로 누른 뒤 돌아갈 창 자체가 없었다.
- `runManagementTurn()`은 즉시 `busy`를 세우고 다시 그린 뒤 엔진을 먼저 계산하고, LLM 서사 또는 `서사화 API 오류 · 엔진 결과 유지` 폴백을 같은 채팅에 추가했다(`:1119-1154`). 모든 현장 버튼은 `busy` 동안 비활성화됐다(`:937-1033`).

**현 동작 (`895efab`)**

- 엔진 계산·자동 서사·실패 폴백은 `PlaySession.runManagementTurn()`으로 복구됐다(`packages/session/src/index.ts:77-78`).
- 그러나 `InnManagement.svelte:15`와 `ScreenRenderer.svelte:22`가 별도 시뮬레이션 패널 안에서 직접 LLM 왕복을 기다린다. 상위 `PlayerPage`는 행동 시작을 모르므로 패널을 닫거나 채팅 대기 표시를 켜거나 버튼을 잠그거나 오류를 토스트로 보여줄 수 없다.

**격차와 판정 — 부분.** 계산과 저장은 동등하지만 사용자가 눌렀다는 피드백과 채팅 복귀가 끊겼다. WP1에서 실행 책임을 `PlayerPage`로 올려야 한다.

### 흐름 3 — 시뮬레이션에서 장부 행동을 누름

**옛 동작 (`03074d8^`)**

- 구매·증축 같은 준비 행동은 LLM을 부르지 않고 즉시 처리했다. 결과를 성공/실패 칩으로 요약하고 다음 현장 서사에 들어갈 `ledgerDeltas`를 쌓았다(`app/src/playView.js:1094-1117`).
- 셀렉터가 계산한 가격과 `affordable`로 버튼을 미리 잠갔다(`engine/core/selectors.js:52-63`, `app/src/playView.js:1040-1089`).

**현 동작 (`895efab`)**

- `PlaySession.runLedgerAction()`은 LLM을 쓰지 않고 사건·원장·다음 장면 문맥을 저장한다(`packages/session/src/index.ts:76`). 이 핵심은 복구됐다.
- 하지만 사용자 메시지는 모든 행동에 같은 `장부에 반영되었습니다.`이고, 현재 UI는 구체적인 성공/실패 이유와 사전 가능 여부를 거의 보여주지 않는다. `InnManagement.svelte:14-15`는 어떤 행동을 장부로 볼지도 UI의 고정 집합으로 결정한다.

**격차와 판정 — 부분.** 장부 행동은 패널을 자동으로 닫지 않는 것이 맞지만, 결과 영수증과 셀렉터 기반 사전 안내를 복구해야 한다. WP1의 현장 오케스트레이션과 WP5의 셀렉터를 분리한다.

### 흐름 4 — AI 응답에 감정 스프라이트가 표시됨

**옛 동작 (`03074d8^`)**

- LLM이 돌려준 화자를 메시지의 `npcIds`에 고정하고(`app/src/playView.js:1142`, `:1184`), 해당 메시지 옆 작은 NPC 아바타로 그렸다(`:560-566`, `:668-679`). 별도의 큰 무대도 있었지만 메시지 이미지의 대체물이 아니었다(`:640-666`).
- 에셋 선택은 `resolveSpeaker`와 현재 엔진 상태를 거쳤다(`:618-637`).

**현 동작 (`895efab`)**

- 메시지별 `speakers` 저장과 대안 전환은 복구돼 있다. `MessageList.svelte:32-33`도 해당 메시지 화자를 읽는다.
- 그러나 `MessageList.svelte:42`가 모든 화자를 큰 `SpeakerStage`로 다시 그리고, 작은 아바타는 첫 화자 하나뿐이다. 본문의 `<img src="silvia_default">`는 `asset-macros.ts:18-33`의 정확 이름 검색을 거쳐 실제로 존재하는 `silvia_default_0` 계열을 찾지 못한다.

**격차와 판정 — 부분.** 데이터는 있으나 화면 표현과 에셋 해석이 불완전하다. WP2+WP3를 하나로 처리해 본문 이미지를 먼저 살리고 큰 메시지 무대를 제거해야 한다.

### 흐름 5 — NPC 의상이 바뀜 (`set_outfit`)

**옛 동작 (`03074d8^`)**

- 고용된 직원의 실제 에셋 변형만 의상 선택지로 보여주고, 바꾸면 장부 사건 `set_outfit`을 실행했다(`app/src/playView.js:1517-1532`).
- 렌더할 때마다 `outfitOf()`로 엔진 상태를 읽어 `selectAsset(group, emotion, outfit)`에 전달했다(`:737`, `:756`).

**현 동작 (`895efab`)**

- 엔진은 `state.npcs[id].outfit`을 저장하고 범위를 검증한다(`packages/modules/src/inn.ts:23`). 런타임 화자 해석기도 이를 읽는다(`packages/runtime/src/speaker.ts:85-100`).
- 마지막 UI 계약이 `portraitFor(id, emotion?)`뿐이라 outfit이 빠진다(`PlayerPage.svelte:120`). `selectNpcSprite(group, emotion)`도 outfit 대신 이름 해시로 변형을 고른다(`npc-gallery.ts:24`, `:34`).

**격차와 판정 — 유실.** 상태와 이미지가 디싱크된다. WP2+WP3에서 정확 파일명 우선 → 실제 outfit 변형 → 카드가 선언한 정책 → 일반 카드의 결정론적 기본값 순서로 공용 resolver를 완성한다. 숫자 접미사를 모든 카드의 의상으로 일반화하면 안 된다.

### 흐름 6 — 직원을 고용/해고함

**옛 동작 (`03074d8^`)**

- 화면에는 현재 직원과 `현재 인원/최대 정원`이 보였다(`app/src/playView.js:1511-1545`; `engine/core/selectors.js:6-14`).
- `직원 모집`은 정원이 남을 때만 활성화되고, 모든 NPC 가격표가 아니라 “고용 후보와 자연스러운 협상 장면”을 채팅으로 열었다. 임금 합의 전에는 `hire`를 내지 말라는 계약이 있었다(`playView.js:918-925`). 해고도 통보 장면을 열었다(`:1536-1537`).

**현 동작 (`895efab`)**

- 엔진의 정원·중복 고용·해고 검사는 살아 있다(`packages/modules/src/inn.ts:8`, `:20-22`). `inn/staff-capacity` 셀렉터도 있지만 UI가 사용하지 않는다(`:39`).
- `InnManagement.svelte:35`는 모든 NPC를 `고용 가능`으로 나열하고 사용자가 임금을 임의 입력해 바로 `hire`한다. 해고 버튼은 없다.

**격차와 판정 — 유실.** 단순 UI 축소가 아니라 엔진 사실과 협상 계약을 우회하는 치트 채널이다. WP5에서 “협상 시작”과 현재 직원·정원·해고 흐름으로 되돌린다.

### 흐름 7 — 시설을 증축함

**옛 동작 (`03074d8^`)**

- 현재 레벨, 다음 레벨, 실제 비용, 최대 여부를 한 줄에 보여주고 돈이 부족하거나 최대면 버튼을 잠갔다(`engine/core/selectors.js:59-64`; `app/src/playView.js:1010-1013`, `:1087-1089`).
- 장부 행동이라 LLM 없이 즉시 결과 칩을 남겼다.

**현 동작 (`895efab`)**

- 엔진은 전투 중·최대 레벨·비용 존재·잔액을 모두 검증하고 실제 골드를 차감한다(`packages/modules/src/inn.ts:19`).
- UI는 레벨만 표시하고 버튼 이름도 `증축`뿐이며 비용·가능 여부·실패 이유를 누르기 전에 보여주지 않는다(`InnManagement.svelte:33`).

**격차와 판정 — 부분.** 계산은 맞지만 사용자는 실패를 예측할 수 없다. WP5의 공용 관리 셀렉터가 `nextCost`, `affordable`, `maxed`를 소유해야 한다.

### 흐름 8 — 의뢰를 수행함

**옛 동작 (`03074d8^`)**

- 주사위 전체 경우를 열거해 실제 성공 확률을 계산했고(`engine/core/selectors.js:86-99`), 화면에 성공률·보상 범위·완료·오늘 시도·진행 중 상태를 표시하고 불가능한 버튼을 잠갔다(`:69-80`; `app/src/playView.js:1029-1035`).

**현 동작 (`895efab`)**

- 엔진은 게시판 해금, 오늘 목록, 전투 조우, 중복 시도, 실제 판정과 보상을 결정론적으로 처리한다(`packages/modules/src/inn.ts:11-15`, `:38-39`).
- UI는 이름과 보상 등급만 표시하고 언제나 `수행` 버튼을 내놓는다(`InnManagement.svelte:39`). 진짜 확률·보상 범위·오늘 처리 여부는 없다.

**격차와 판정 — 부분.** 실행 결과는 맞지만 행동 전 정보가 유실됐다. WP5에서 엔진과 같은 수학을 공유하는 셀렉터를 복구한다. UI가 확률을 다시 계산해서는 안 된다.

### 흐름 9 — 시뮬레이션 화면을 열고 닫음

**옛 동작 (`03074d8^`)**

- 데스크톱 관리 기능은 채팅 안의 `현장 행동` 접이식 메뉴였으므로 별도 시뮬레이션 화면을 열고 닫지 않았다(`app/src/playView.js:908-917`).
- 모바일 상태·설정은 바텀시트였고 명시적 닫기, 바깥 클릭, ESC, 포커스 복귀를 지원했다(`:381-416`).

**현 동작 (`895efab`)**

- `SimulationPanel.svelte:9-11`은 명시적 닫기 버튼이 있고 공용 `OverlayGuard`를 통해 ESC도 지원한다. 이전의 보이지 않는 close 아이콘 문제는 이미 해결됐다.
- 다만 데스크톱에서도 채팅 옆 별도 도킹 패널이고, 모바일은 전체화면이다(`SimulationPanel.svelte:16-18`). 화면 제목 `여관 경영`은 내비게이션 버튼과 본문 제목으로 두 번 보인다(`ScreenRenderer.svelte:28-30`). 현장 행동 뒤 자동 복귀도 없다.

**격차와 판정 — 부분.** 열고 닫는 접근성은 복구됐지만, 별도 패널로 바꾼 데 따른 행동 후 복귀와 단일 제목 계약이 빠졌다. WP1과 WP6에서 나눠 처리한다.

## 4. 추가로 확인된 유실과 범위 경계

WP0에서 10개 흐름 밖의 다음 사항도 확인했다.

1. **셀렉터 유실은 여관 화면보다 넓다.** 옛 `availableActions()`는 전투 스킬 코스트·사용 가능 여부·도주 확률을, `usableItems()`는 소모품 사용 가능 여부를 제공했다. 현행 UI에 동등한 공용 어포던스 계층이 없다. WP5보다 먼저 여관을 고치되, 여관 전용 계산을 UI에 박지 않고 확장 가능한 selector API로 복구해야 한다.
2. **방 정보도 축소됐다.** 현행 `inn/rooms`는 잠김과 투숙객은 주지만 옛 `roomStatus()`의 정원·빈자리 계약 전체를 직접 제공하지 않는다. 숙박 수락 화면의 사전 판단까지 WP5 감사 범위에 포함한다.
3. **현장과 장부의 경계가 UI 상수에 중복됐다.** `InnManagement.svelte:14`의 `LEDGER_EVENTS`는 사건 의미를 UI가 다시 분류한다. 선언형 화면의 `mode:'ledger'`와 장르 모듈 메타데이터 중 한 곳이 정책을 소유하도록 WP1에서 경계를 정해야 한다.
4. **프롬프트의 조건 분기는 별도 확정 결함이다.** `postHistoryInstructions`는 프롬프트 컴파일러로 들어가지만(`packages/risu/src/compiler.ts:11`), `parseCbs()`는 `#when`/`module_enabled` 문맥을 지원하지 않는다(`packages/risu/src/cbs.ts:16`). 표시용 CBS만 고쳐서는 해결되지 않으며 WP4 범위를 유지한다.
5. **큰 무대를 전역 삭제하면 안 된다.** 메시지마다 자동 삽입되는 `SpeakerStage`만 제거 대상이다. SimPack이 명시적으로 선언한 `speaker-stage` 위젯은 제작자 의도이므로 유지한다(`ScreenRenderer.svelte:43`).

## 5. WP1~WP7 조정안

WP0 결과로 큰 순서는 유지하되 범위를 다음처럼 확정한다.

1. **WP1 — 시뮬레이션 행동 오케스트레이션.** 현장 행동만 상위 `PlayerPage`에 요청하고 즉시 패널 닫기 → 채팅 대기 표시 → 최신 사용자/대기 위치 이동 → 응답 도착 위치 이동을 묶는다. 장부 행동은 패널에 남되 구체 결과를 표시한다. `LEDGER_EVENTS` UI 상수의 정책 소유 위치도 정리한다.
2. **WP2+WP3 — 공용 에셋 resolver와 메시지 표현.** 분리하지 않는다. `outfit`을 끝까지 전달하고 실물 `silvia_default_{N}` 회귀 테스트를 만든 뒤, 메시지 자동 큰 무대를 작은 화자 아바타로 바꾼다. 명시적 `speaker-stage` 위젯은 유지한다.
3. **WP4 — 관리 서사 프롬프트와 안전 조건 평가.** `YSP_default` 반복을 프롬프트·첫 메시지·regex 단계별로 분리해 원인을 증명한다. 표시와 프롬프트가 공유하는 `#when`/`module_enabled` 평가기를 만들되 카드 이름 분기는 금지한다.
4. **WP5 — 관리 어포던스 selector 복구.** 여관의 정원·협상·해고·시설 비용·확률·숙박 빈자리·영업 순서·사건 대기·구매 합계를 먼저 복구한다. 계산은 UI가 아니라 selector가 소유한다. API는 이후 전투 `availableActions`와 소모품까지 확장할 수 있게 설계하되, 이번 WP에서 범위를 무리하게 넓히지 않는다.
5. **WP6 — 화면 컨테이너와 단일 제목.** 시뮬레이션 패널과 편집기 미리보기가 각각 자기 폭에 반응하도록 한다. 내비게이션 항목이 하나뿐이면 중복 탭을 숨기고 화면 제목은 한 번만 보여준다.
6. **WP7 — 구현 후 사용자 여정 확대 감사.** 전투 선택지·소모품·구매/판매·숙박 일괄·사건 대응·우편·하루 마감까지 “무엇이 보이고 어디로 이동하는가” 기준으로 재검증한다. 여기서 나온 결함은 별도 후속 WP로 분리한다.

고정 순서는 유지한다: **WP1 → WP2+3 → WP4 → WP5 → WP6 → WP7**. WP2와 WP3는 한 커밋 단위이며, WP5의 selector 기반 없이 전투·소모품 UI를 먼저 만들지 않는다.

## 6. WP0 완료 범위

## 8. 확대 감사의 판정 기준

엔진에 사건 이름이 있다는 것만으로 사용자 기능이 완성된 것은 아니다. 앞으로는 다음 세 층을 따로 판정한다.

1. **엔진 경로**: 사건을 직접 호출하면 결정론 계산과 검증이 작동한다.
2. **선언형 경로**: 제작자가 `action-group` 같은 화면을 직접 작성하면 사건을 호출할 수 있다.
3. **기본 사용자 여정**: 컴파일된 카드의 기본 화면에서 사용자가 필요한 정보와 버튼을 보고, 결과를 확인한 뒤 다음 행동으로 이동할 수 있다.

세 번째 층까지 연결되어야 플레이 기능을 “구현 완료”로 적는다. 현재 내장 모듈 17개에는 사건 60개와 selector 21개가 등록되어 있지만 process는 0개다. `ScreenRenderer`의 전용 상호작용 컴포넌트는 `chat`, `inn-management`, `action-group`, `decision-card`, `speaker-stage` 다섯 종류다. `card-list`, `map-nodes`, `inventory-grid`, `quest-board`는 공용 선택 스킨이며, 나머지 위젯 이름은 구조화 데이터 폴백으로 표시될 수 있을 뿐 전용 사용자 흐름을 증명하지 않는다.

## 10. 후속 작업 권고

1. **전투 콘솔과 소모품**: 전투 selector를 사용자 어포던스로 확정하고 시작·행동·적 턴·종료·아이템 사용을 한 화면과 e2e 여정으로 연결한다. 의뢰가 전투를 시작했을 때 막히지 않는 것이 최우선 완료 조건이다.
2. **여관 상거래·채집·숙박 일괄**: 옛 화면의 비용·가능 여부를 selector가 소유하게 하고, `runManagementBatch()`를 안전한 일괄 처리 버튼에 연결한다.
3. **헌터 카나리아 화면**: 전투 콘솔 위에 등록·평가·파티·게이트·정산을 연결한다. 카드 이름 분기 없이 장르 모듈과 선언형 화면만으로 완주해야 한다.
4. **결과 영수증 확대**: 전투·소모품·상거래 사건의 실제 증감을 사람이 이해할 수 있는 문구로 표시한다.
5. **위젯·편집기 레지스트리 정직화**: 장비·상점·제작·직업·위치·파티는 전용 UI가 생길 때만 구현 완료로 승격하고, 문서 이름이나 generic 폴백을 완료 근거로 삼지 않는다.

각 후속 작업은 “버튼이 존재한다”가 아니라 시작 전 정보 → 실행 중 피드백 → 결과 확인 → 다음 화면 이동을 브라우저 e2e로 증명해야 한다.
