# 리스 플러그인 장기 기억 이식 감사 (2026-07-19)

## 한줄 결론

장기 기억의 **저장·근거 검증·검색·되돌리기·프롬프트 재주입 골격은 현재 프로젝트에 남아 있고 테스트도 통과한다.** 그러나 일반 채팅에서 모델에게 기억 후보를 매번 제출하라고 요구하는 기본 출력 계약이 빠져 있어, 실제 플레이에서는 엔진 사실과 일부 관계 기록만 잘 쌓이고 순수 서사 기억은 거의 수집되지 않을 수 있다. 따라서 판정은 **“기반 보존, 실사용 수집은 부분 적용”**이다.

## 사용자 화면에서의 의미

- 돈·아이템·관계 티어처럼 엔진이 확정한 사실은 장기 회차에도 기억 원장에 남는다.
- 되돌리기나 다른 응답으로 바꾸면 그 이후에 생긴 기억도 함께 되돌아간다.
- 모델이 근거 없이 만든 기억은 바로 확정되지 않고 `세션 검사 → 기억 검토`에 대기한다.
- 관련 기억은 다음 대화에 최대 8건, 약 1,000토큰 범위로 들어가므로 대화가 길어져도 프롬프트가 무한히 커지지 않는다.
- 반면 “어제 둘만의 비밀을 털어놓았다”, “다음 주에 만나기로 약속했다”처럼 엔진 사건이 아닌 서사 사실은 모델이 구조화된 `memories`를 스스로 내지 않으면 저장되지 않는다. 현재 기본 프롬프트에는 이를 요구하는 문장이 없다.

## 실제로 잘 남아 있는 부분

1. **엔진이 사실을 소유한다.** 모델이 돈·체력·관계를 기억한다고 주장하는 대신 성공한 사건 로그가 기억의 근거가 된다. 제품 ADR의 금지선과 일치한다.
2. **거짓 기억이 바로 확정되지 않는다.** 모델 단독 주장은 후보로만 남고 사용자가 승인해야 한다. 원본 플러그인의 편리함보다 안전성을 우선한 의도적인 변형이다.
3. **시간 여행이 기억까지 함께 움직인다.** undo, redo, reroll, 응답 대안, 사건 잘라내기에서 미래 기억이 남지 않는 테스트가 있다.
4. **검색 결과가 제한돼 있다.** 최근 대화 전체를 계속 늘리는 대신 관련 기록만 고르고, 검색 실패 시 추측하지 않도록 한다.
5. **저장 후 다시 열어도 유지된다.** 기억, 검토 대기, 연속성 변경 제안이 세션 저장과 무결성 검사에 포함된다.

## 가장 중요한 누락

### 1. 기억 수집 지시가 기본 요청에 없다

원본 HAYAKU는 매 응답에 `HAYAKU_STATE_PACKET`을 쓰도록 출력 계약을 주입하고, LIBRA는 유지보수 단계에서 기억을 추출한다. 현재 프로젝트는 `memories`, `factRefs`, `continuityPatch`를 **받아 처리하는 파서와 안전장치만** 있고, 일반 모델 요청에 그 필드를 작성하라는 계약을 넣지 않는다. 테스트는 가짜 provider가 해당 필드를 직접 반환하므로 이 공백을 발견하지 못한다.

권장 후속: 사용자 프리셋 내용과 독립된 작은 시스템 계약으로 “중요한 약속·비밀·관계 변화만 후보로 제출하고 근거 인용/사건 ID를 붙인다”를 추가한다. 기존 승인 관문은 그대로 유지하므로 무검수 자동 승인이 되지 않는다.

### 2. 인물별로 아는 정보가 완전히 분리되지 않는다

비밀·소유자·차단 대상 필드는 있지만 검색할 때 현재 화자 목록을 넘기지 않는다. 따라서 “A만 아는 비밀”을 A에게만 주입하는 LIBRA식 POV는 완성되지 않았다.

권장 후속: 현재 장면 화자와 시점 인물을 selector에서 얻어 `viewer.entityIds`에 전달하고, `visibleToEntityIds`도 필터에 반영한다. 서로 다른 인물이 같은 사건을 다르게 아는 회귀 테스트가 필요하다.

### 3. 오래된 서사 기억의 보관 정책이 없다

반복 엔진 사실은 이미 정리하지만, 약속·에피소드·요약 후보는 계속 한 배열에 남는다. 기능 유실은 아니지만 수천 턴 뒤 저장 크기와 검색 비용이 늘 수 있다.

권장 후속: 원문 사건은 저널에 남기고, 오래된 서사 기억은 근거 링크를 유지한 요약·페이지 보관으로 옮긴다. 이는 별도로 기록한 장기 회차 성능 문제와 함께 다루되 원본 백업과 무손실 이주를 관문으로 둔다.

### 4. 실모델 장기 기억 품질은 아직 증명되지 않았다

현재 벤치마크는 300턴·120문항 고정 fixture와 결정론 임베딩으로 검색 파이프라인을 검증한다. 실제 모델이 올바른 기억 후보를 꾸준히 작성하는지, Voyage가 실제 한국어 의역 회상을 개선하는지는 라이브 장기 시험이 남아 있다.

## 플러그인별 최종 판정

- **HAYAKU:** 롤백 정합, 근거, 현재/과거, 비밀 경계의 철학은 남았다. 하지만 핵심인 매턴 상태 패킷 수집과 정밀 locator·페이지 아카이브는 완전히 이식되지 않았다.
- **LIBRA:** 엔진 근거, 승인 관문, 비밀 스키마, 선택적 임베딩은 남았다. 자동 유지보수·TTL·세계/인물 지식 투영·POV는 일부 또는 미이식이다.
- **RisuAI Agent:** 최신성 가중과 임베딩 배치·캐시 아이디어는 현재 검색기에 반영됐다. 로어북을 직접 고치는 에이전트 구조는 가져오지 않았다.
- **Agents!:** 현재 세션 체크포인트가 비슷한 롤백 안전성을 제공하지만, 에이전트별 기억 스택 자체는 이식하지 않았다.
- **MultiAgent Full:** 장기 기억 원본이 아니라 최근 문맥 분석 파이프라인이므로 이번 장기 기억 감사의 직접 비교 대상이 아니다.

## 검증 결과

- `@simbot/memory`: 7개 테스트 파일, 29개 테스트 통과. 기억 승인·비밀 범위·현재/과거 교체·검색 포기·Voyage 안전 어댑터·300턴 fixture가 정상임을 확인했다.
- `@simbot/session`: 27개 테스트 파일, 146개 테스트 통과. 기억과 엔진 상태의 저장·복원·undo/redo·대안 응답·300턴 프롬프트 상한이 함께 유지됨을 확인했다.
- 테스트가 증명하지 못하는 것: 실제 모델의 기억 후보 생성률, 실제 한국어 장기 회상 품질, 수천 턴 서사 기억의 저장 비용.

## 권장 작업 순서 (이번 감사에서는 구현하지 않음)

1. 기본 응답에 기억 후보 출력 계약 연결.
2. 현재 화자 기반 POV·비밀 필터 완성.
3. 실제 모델로 300턴 수집률·회상 정확도 측정.
4. 서사 기억 요약·페이지 보관과 기존 회차 무손실 이주.
5. 그 뒤에만 Voyage 기본 활성화 여부를 결정.
